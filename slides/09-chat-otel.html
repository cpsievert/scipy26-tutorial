Observability

Monitoring chatlas apps with OpenTelemetry

Shipping raises new questions

Your chatbot works on your machine. Then it ships, and:

  • Why is it slow for some users?
  • Which model calls or tool executions dominate latency?
  • Where are errors coming from?
  • What are people even asking it?

This is what OpenTelemetry (OTel) is for.

OpenTelemetry, briefly

  • A vendor-neutral, open standard for collecting traces, metrics, and logs from an app.
  • Instrument once, view anywhere – Logfire, Datadog, Honeycomb, Jaeger, and others all speak OTel.
  • chatlas is instrumented with OTel out of the box – no tracing code to write yourself, just choose where to send the data.

Two concepts: span & trace

  • A span is one timed unit of work – one model call, one tool execution – with a name and key/value attributes.
  • A trace is a tree of spans: the full path of one request, each nested under the one that triggered it.

A single chat() that calls a tool produces a trace shaped like this:

invoke_agent                      # wraps the full chat loop
├── chat gpt-4o                   # each model API call
├── execute_tool get_weather      # each tool invocation
├── chat gpt-4o                   # follow-up model call
└── ...

Quick start: console output

pip install "chatlas[otel]"


from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import ConsoleSpanExporter, SimpleSpanProcessor

provider = TracerProvider()
provider.add_span_processor(SimpleSpanProcessor(ConsoleSpanExporter()))
trace.set_tracer_provider(provider)
  • That’s it – chatlas now emits spans for every chat(), stream(), and tool call.

Worked example: a tool with its own span

pip install opentelemetry-instrumentation-httpx


travel_assistant.py
from opentelemetry import trace
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor
from chatlas import ChatBedrockAnthropic

HTTPXClientInstrumentor().instrument()  # nests the HTTP call under `chat`
tracer = trace.get_tracer("travel_assistant")

def get_weather(city: str) -> str:
    "Get the current weather forecast for a city."
    with tracer.start_as_current_span("fetch_forecast"):
        return f"{city}: 14°C, light rain, breezy this weekend."

chat = ChatBedrockAnthropic(
    system_prompt="Always call get_weather before advising what to pack."
)
chat.register_tool(get_weather)
chat.chat("I'm headed to Tokyo this weekend — what should I pack?")

The resulting trace

  • Read top to bottom: the model decides to call the tool, the tool runs (fetch_forecast nested inside execute_tool), and a follow-up model call answers.
  • Drop the httpx instrumentation and you’d still get invoke_agent, chat, and execute_tool – just without the HTTP detail.

What’s captured

Span Captures
invoke_agent – the whole chat loop Provider, requested model
chat – one per model call Provider/model, token usage, response id
execute_tool – one per tool call Tool name, description, call id, errors
  • Attribute names follow the OTel GenAI semantic conventions (gen_ai.*) – the same names other GenAI-instrumented libraries use.

Message content is opt-in

  • Message content – the actual prompts and responses – is not captured by default.
  • It may contain sensitive data.
export OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT=true
  • Your users’ prompts stay out of your traces unless you choose otherwise.

Wiring it into a Shiny app

otel_config.py
import logfire

app.py
import otel_config  # noqa: F401 -- side-effect import, runs before anything else
from chatlas import ChatBedrockAnthropic
# ...
  • The exporter has to be configured before chatlas creates any chats – a dedicated module you import first guarantees that.

Config without code changes

OTEL_EXPORTER_OTLP_ENDPOINT="https://logfire-eu.pydantic.dev"
OTEL_EXPORTER_OTLP_HEADERS="Authorization=<your-write-token>"
OTEL_SERVICE_NAME="my-chatlas-app"
  • Same app, same code – point it at a different backend, or a different environment, by changing env vars, not source.
  • On Posit Connect, set these as content environment variables – no redeploy needed to change where traces go.

Going deeper (optional)

chatlas’s spans describe the shape of a conversation. To see inside each step – raw HTTP, retries, full payloads – add a lower-level instrumentor:

Level Example Tradeoff
Transport httpx instrumentation Works with every provider; generic HTTP spans only
SDK, model-agnostic OpenLLMetry GenAI-aware spans across many providers
Official, per-provider e.g. opentelemetry-instrumentation-anthropic Most detail; one package per SDK
  • Start with httpx (you just saw it) – reach for the others only if you need GenAI-specific attributes chatlas doesn’t already give you.

Don’t dwell here – 30 seconds max. The point is just that more detail is available if they need it; most people in this room will never need more than the framework-level spans chatlas gives them for free.
