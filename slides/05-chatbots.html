shinychat

Chatbots made easy

Turn a chatlas client into a full chatbot UI — in 5 lines.

shinychat is the bridge between “I can call an LLM in a script” and “I have an app other people can use.” It’s built on Shiny for Python, but you don’t need to know Shiny to use it.

The 5-line chatbot

app.py
from chatlas import ChatBedrockAnthropic
from shinychat.express import Chat

client = ChatBedrockAnthropic(system_prompt="You are a helpful assistant.")
chat = Chat(id="chat", client=client)
chat.ui()
shiny run app.py

Live demo: create this file, run it, open the browser. This is the wow moment. By passing client=, shinychat automatically wires up: - Streaming responses - Multi-turn conversation history - Cancel button during generation - File attachment support - Bookmarkable state

All of that from 5 lines. Let them absorb this before moving on.

What client= gives you for free

  • ✅ Streaming responses
  • ✅ Multi-turn conversation
  • ✅ Tool calling with display
  • ✅ Cancel mid-generation
  • ✅ File attachments (images, PDFs)
  • ✅ Bookmarkable state
  • ✅ Automatic error handling
  • ✅ Async by default

You could wire all this up manually… but why would you?

The point of this slide is to justify the abstraction. Each of these features would be dozens of lines of code to implement from scratch. The client= parameter is doing a lot of work under the hood — it calls client.stream_async(user_input, *attachments, content="all") and feeds the result into the chat UI.

Bring your own backend

Not using chatlas? Wire up any async generator:

app.py
from shinychat.express import Chat

chat = Chat(id="chat")
chat.ui()

@chat.on_user_submit
async def handle_user_input(user_input: str):
    response = my_langchain_chain.astream(user_input)
    await chat.append_message_stream(response)

Brief slide — just establish the escape hatch. If they’re using LangChain, Pydantic AI, or any other framework, they can still use shinychat as the UI layer. The @chat.on_user_submit decorator gives you full control over what happens when the user sends a message.

Don’t spend more than 60 seconds here — most attendees will use the chatlas path.

Custom tool displays

By default, tool results show as collapsible text. But you can make them beautiful.

SCREENSHOT: side-by-side comparison —
(left) default tool result display (plain collapsible text)
(right) custom ToolResultDisplay with icon, title, and formatted markdown

This is the main teaching point of the shinychat section. When a chatlas tool runs, the result shows up in the chat as a collapsible block. That’s fine for debugging, but for end users you want something polished. ToolResultDisplay lets you control exactly how it looks.

Show the default first, then show the custom version. The contrast is the motivator.

Building a tool display

from chatlas import ContentToolResult
from shinychat.types import ToolResultDisplay
import faicons

def get_weather(lat: float, lng: float, location: str) -> ContentToolResult:
    """Get the current weather for a location.
    ...
    """
    data = call_weather_api(lat, lng)
    return ContentToolResult(
        value=data,                                  # what the LLM sees
        extra={
            "display": ToolResultDisplay(
                title=f"Weather: {location}",
                icon=faicons.icon_svg("cloud-sun"),
                markdown=f"**{data['temp']}°C** — {data['condition']}",
            )
        },
    )

Walk through the anatomy:

  1. Return type is ContentToolResult, not a plain value.
  2. value= is what the LLM sees — it still needs the raw data to reason about.
  3. extra={"display": ToolResultDisplay(...)} is what the user sees in the chat UI. The LLM never sees this.
  4. title and icon appear in the collapsed header. markdown (or html or text) is the expanded body.

Other useful params: full_screen=True adds an expand button (great for charts/tables), open=True starts expanded, footer for attribution.

Observability with OpenTelemetry

chatlas emits OTel spans for every LLM call — latency, tokens, errors.

# In production, configure your OTel exporter before the app starts
# e.g., with Logfire, Datadog, Honeycomb, etc.
import logfire
logfire.configure()

# Then just run your shinychat app as normal — tracing is automatic

See what your users are actually asking, how long it takes, and what fails.

Brief slide. OTel support is built into chatlas, so shinychat apps get it for free. In production you’d configure an exporter (Logfire is the easiest if you’re in the Posit ecosystem). This gives you dashboards showing: what questions users ask, how long responses take, token usage, error rates.

Don’t go deep — just plant the idea that observability is built in, not bolted on.
